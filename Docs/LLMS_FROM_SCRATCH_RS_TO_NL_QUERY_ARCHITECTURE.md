## Architecture snapshot to keep in mind

Before reading the chapter material, keep the target architecture in view. The destination system is not “train a GPT clone and make it chat.” It is a single-binary Rust application that uses local model inference as one bounded part of a deterministic retrieval pipeline.

Key facts from the target design:

| Area | Target architecture fact |
|---|---|
| Runtime | One Rust process owns Axum, Dioxus SSR rendering, embedded SurrealDB, classifier state, embedding model state, and refresh logic. |
| Query routing | A DeBERTa v3-small sequence classifier predicts one of `graph_search`, `vector_search`, or `graph_rag`. |
| Embeddings | A local BGE-small-style embedding model produces 384-dimensional vectors for semantic search. |
| Query generation | SurrealQL is selected by deterministic Rust templates, not generated by an LLM. |
| Graph RAG | Phase 11 is retrieval-only: graph and vector arms run, merge, deduplicate, rank, and stream structured context. No LLM answer synthesis is active in that phase. |
| Frontend | Server-rendered pages use small JS islands for SSE streaming. |
| Data/privacy | Healthcare-like synthetic data stays local; external LLM APIs are out of scope for the active retrieval path. |
| Performance | The architecture has explicit latency budgets: first SSE event under 50 ms, graph/vector paths under 500 ms, full Graph RAG context under 2 s. |

That means this repo transfers in a non-obvious way. The book and Rust port are GPT/decoder-heavy and training-oriented. The target app is encoder-heavy, inference-oriented, and retrieval-first. The right learning move is **not** to copy the GPT implementation directly. The right move is to learn the Rust/`candle` model substrate: tensors, tokenizers, datasets, attention, module composition, classifiers, checkpoints, collators, and parameter selection.

## Rust and candle patterns to watch for

### `Tensor`, `Device`, and explicit fallibility

In Python, tensor code often looks like ordinary math until something fails at runtime. In `candle`, many operations return `Result<Tensor>`, so shape/device mistakes are visible in function signatures and handled with `?`.

Common operations you will see repeatedly:

- `Tensor::new(...)`
- `Tensor::from_vec(...)`
- `reshape(...)`
- `flatten(...)` / `flatten_all(...)`
- `transpose(...)`
- `contiguous()`
- `broadcast_add(...)`
- `broadcast_mul(...)`
- `index_select(...)`
- `matmul(...)`

The repo often uses `Device::cuda_if_available(0)?`. Treat that as a teaching convenience, not a universal deployment rule. The target architecture’s CPU/GPU decision is governed by measured latency, parity, memory, and lifecycle behavior.

### `VarMap` and `VarBuilder`

If you come from PyTorch, a rough mental map is:

| Python/PyTorch instinct | Rust/`candle` equivalent here |
|---|---|
| model `state_dict` / named parameter store | `VarMap` |
| scoped parameter creation | `VarBuilder` |
| submodule path prefix | `vb.pp("...")` |
| model checkpoint artifact | `VarMap::save(...)` / `VarMap::load(...)` |

Parameter names matter. In later chapters, weight loading and selective fine-tuning depend on names such as `model.trf.11`, `final_norm`, `out_head`, and LoRA adapter names containing `A` or `B`.

### `Module` vs `ModuleT`

`Module` is the basic forward-pass trait. `ModuleT` is the training-aware variant with a `train` flag, needed for layers like dropout.

A common beginner mistake is to treat `forward(...)` and `forward_t(..., train)` as interchangeable. They are not. If a layer behaves differently during training and inference, the `train` flag matters.

### Traits as architecture seams

The repo uses traits as small protocol boundaries:

- [`Example`][sym-example-trait] and [`Exercise`][sym-exercise-trait] in [`src/lib.rs`][src-lib]
- [`DataLoader`][sym-dataloader-trait] in [`src/listings/ch02.rs`][ch02-listings]
- [`GPT`][sym-gpt-trait] in [`src/listings/ch04.rs`][ch04-listings]
- [`PromptFormatter`][sym-promptformatter] and [`CustomCollator`][sym-customcollator] in [`src/listings/ch07/mod.rs`][ch07-listings]

For a Python reader, think “interface/protocol,” but with compile-time checking.

### Dataset, iterator, collator, loader

Several chapters repeat this pattern:

1. dataset stores examples,
2. iterator yields one example at a time,
3. collator turns variable examples into rectangular tensors,
4. loader/batcher yields batches to training/evaluation loops.

This is one of the strongest direct bridges to the target architecture. Real model systems spend a surprising amount of effort making inputs boring, consistent, and inspectable.

### Why this chapter matters

Chapter 1 builds the mental model that keeps later code honest. Without it, two shortcuts become tempting: treating an LLM as “basically a chatbot,” and treating every transformer as “basically GPT.” Both are too sloppy for architecture work.

For this repo, Chapter 1 explains what kind of model the later Rust code builds: a decoder-only GPT-style transformer trained for next-token prediction. In this guide, the book’s “LLM” is primarily that autoregressive GPT-style model; the target architecture also uses encoder-style transformer-derived models whose outputs are class logits or embeddings rather than next-token probabilities. For the target architecture, Chapter 1 helps you decide what to transfer and what not to over-map. The target system is retrieval-first and encoder-heavy, not a straight GPT clone.

### Book anchors and learning notes

#### 1.1 What is an LLM?

A large language model is a learned function from token sequences to probability distributions over token continuations. It is not a database, a rule engine, or a guarantee of truth. It is excellent at fluent pattern completion and weak when hard constraints, deterministic behavior, or factual grounding are required.

That distinction is central to the target architecture. The NL-query system does not ask one model to be the whole application. It uses specialized model components and deterministic retrieval logic.

#### 1.2 Applications of LLMs

LLM-family systems can generate, classify, embed, summarize, translate, rank, or route. Always ask what the model is actually doing. In the target architecture, local transformer-derived components are used mostly for routing and semantic embeddings, not free-form answer generation.

#### 1.3 Stages of building and using LLMs

The lifecycle is roughly: prepare data, tokenize, pretrain, adapt/fine-tune, serve, evaluate. This repo mirrors that progression across Chapters 2–7. The target architecture splits that lifecycle: training/export happen offline; Rust/`candle` inference and retrieval happen in the runtime process.

#### 1.4 Introducing the transformer architecture

The transformer stack converts token IDs into embeddings, mixes information with attention, refines it with feed-forward layers, and projects to task outputs. Later repo anchors include [`MultiHeadAttention`][sym-multiheadattention], [`TransformerBlock`][sym-transformerblock], and [`GPTModel`][sym-gptmodel].

#### 1.5 Utilizing large datasets

Scale comes from parameters, data, and compute. This repo uses small teaching data so you can see mechanics clearly. In the target architecture, the router is specialized via task data, the embedder inherits upstream pretraining, and overall behavior is further constrained by deterministic retrieval and deployment policy.

#### 1.6 A closer look at the GPT architecture

GPT is decoder-only and causal: each position can only see earlier positions. That is essential for generation, but not universal transformer behavior. Encoder-style classifiers and embedding models generally use bidirectional context, meaning a token representation can use tokens on both its left and right rather than only its left-side prefix. Learn GPT deeply, but do not assume GPT is the target runtime shape.

#### 1.7 Building a large language model

Building an LLM means preserving contracts across stages: tokenizer, IDs, shapes, masks, block composition, objective, checkpoint format, and inference policy. That contract discipline transfers directly to the target system even when the exact GPT code does not.

### Rust/`candle` translation

- expect model state to live in `VarMap`,
- expect task behavior to be determined by data format and output heads,

### Why this chapter matters

Chapter 2 is where language modeling becomes operational. Every transformer pipeline, whether it powers generation, classification, reranking, or semantic search, must solve the same first problem: turn text into numbers.

This transfers directly to the target architecture. DeBERTa classification and BGE-style vector search both begin with text normalization, tokenization, IDs, later attention/padding masks, embeddings, and sequence budgets. Chapter 2 itself teaches the pre-mask substrate; encoder-style input packaging adds tokenizer-provided attention masks later.

### Book anchors and learning notes

#### 2.1 Understanding word embeddings

Embeddings are learned lookup-table rows. Given a token ID, the model retrieves a dense vector from a matrix shaped like $[vocab\_size, emb\_dim]$. The vector is not meaningful at random initialization; training or pretrained loading gives it structure.

#### 2.2 Tokenizing text

Tokenization defines the model’s input language. The repo starts with toy tokenizers in [`SimpleTokenizerV1`][sym-simpletokenizerv1] and [`SimpleTokenizerV2`][sym-simpletokenizerv2] so you can see the mechanics. [`sample_create_vocab()`][sym-sample-create-vocab] is a teaching aid built from simple splitting/sorting, not a real tokenizer-training algorithm. Do not mistake those toy tokenizers for production practice.

#### 2.3 Converting tokens into token IDs

Token IDs are lookup keys, not scalar features. `HashMap<String, i32>` in the toy tokenizer is the Rust analogue of a Python dictionary mapping token strings to integers.

#### 2.4 Adding special context tokens

Special tokens such as `<|unk|>` and `<|endoftext|>` encode boundaries and exceptional cases. In production models, special tokens are part of the tokenizer/model contract. Changing them casually changes model behavior. Repo-specific caveat: the toy [`SimpleTokenizerV2`][sym-simpletokenizerv2] is educational and contains a small end-of-text string-literal check quirk, so treat it as a concept demo rather than production tokenizer code.

#### 2.5 Byte pair encoding

BPE moves from toy whole-token splitting to subword units. The repo uses `tiktoken_rs::get_bpe_from_model("gpt2")` in the real dataset path. [`X1`][x-ch02-1] in [`src/exercises/ch02.rs`][ch02-exercises] introduces BPE behavior on unfamiliar text; to reproduce the book’s token-by-token decomposition exactly, add a loop that decodes each token ID individually.

#### 2.6 Data sampling with a sliding window

A long token stream becomes supervised next-token training data through sliding windows. If inputs are `[t0, t1, t2]`, targets are `[t1, t2, t3]`. `max_length` controls window size; `stride` controls how far the window moves.

The key implementation is [`GPTDatasetV1::new(...)`][sym-gptdatasetv1], which tokenizes text and builds input/target token windows. This teaching implementation assumes the text is long enough to produce at least one window; production loaders usually make short-input handling and truncation policy explicit.

#### 2.7 Creating token embeddings

[`EG03`][eg-ch02-03] in [`src/examples/ch02.rs`][ch02-examples] creates an embedding layer with `candle_nn::embedding(...)`. This is the first visible bridge from integer IDs into learnable dense vectors. The example uses randomly initialized weights, so it teaches structure and shape, not already-trained semantics.

#### 2.8 Encoding word positions

Token embeddings encode what token appears; positional embeddings encode where it appears. [`EG04`][eg-ch02-04] adds token embeddings shaped like `[batch, seq_len, emb_dim]` to positional embeddings shaped like `[seq_len, emb_dim]` with `broadcast_add(...)`, producing `[batch, seq_len, emb_dim]`. Chapter 4 repeats this pattern in [`GPTModel`][sym-gptmodel] at model scale.

### Rust/`candle` translation

Key translations:

| Python instinct | Rust/`candle` in this repo |
|---|---|
| `list` | `Vec` |
| `dict` | `HashMap` |
| cheap shared dataset handles | `Rc` |
| generator-style dataset traversal | `Iterator` |
| `torch.Tensor` | `candle_core::Tensor` |
| `nn.Embedding` | `candle_nn::embedding(...)` |

[`GPTDatasetV1`][sym-gptdatasetv1] wraps its backing storage in `Rc`, so cloning a dataset handle is cheap shared ownership, not deep copying. [`GPTDatasetIter`][sym-gptdatasetiter] is where ordinary Rust vectors become `Tensor` values. In this repo, the iterator also chooses the device internally with `Device::cuda_if_available(0)`, which is convenient for examples but usually too implicit for production data loaders.

### Pitfalls

- Confusing tokens with words.
- Treating token IDs as meaningful numeric magnitudes.
- Assuming embeddings are meaningful before training or pretrained loading.
- Forgetting that special tokens are part of the model contract.
- Confusing `max_length` with `stride`.

### Why this chapter matters

Chapter 3 is where token vectors begin to communicate. Attention lets a token decide which other tokens matter and blend their information into a contextual representation.

This matters for the target architecture because DeBERTa-style classifiers and BGE-style embedders are transformer models. They may not use GPT’s causal mask, but they rely on self-attention to make query representations context-sensitive.

### Book anchors and learning notes

#### 3.1 The problem with modeling long sequences

Long-distance dependencies are hard if information has to be compressed or passed step by step. Attention shortens the path: a token can directly score and read other tokens.

#### 3.2 Capturing data dependencies with attention mechanisms

Attention is weighted lookup. Queries ask what is needed, keys advertise what positions contain, and values carry content. Softmax turns compatibility scores into weights.

#### 3.3 Attending to different parts of the input with self-attention

Self-attention uses one sequence as query/key/value source. The output at each position becomes contextual rather than isolated.

##### 3.3.1 A simple self-attention mechanism without trainable weights

[`EG01`][eg-ch03-01] in [`src/examples/ch03.rs`][ch03-examples] shows dot products, normalization, and a weighted sum with tiny tensors. This is concept-first, not production implementation.

##### 3.3.2 Computing attention weights for all input tokens

[`EG02`][eg-ch03-02] computes the full attention matrix. Each row is one token’s distribution over the sequence. Understanding this table is the fastest path to understanding transformer internals.

#### 3.4 Implementing self-attention with trainable weights

Trainable query/key/value projections let the model learn task-useful relevance spaces instead of using raw embedding similarity.

##### 3.4.1 Computing the attention weights step by step

[`EG03`][eg-ch03-03] creates `w_query`, `w_key`, and `w_value`, computes scaled attention scores, and forms a context vector. The scale factor $\sqrt{d_k}$ prevents softmax from becoming too sharp as dimensions grow.

##### 3.4.2 Implementing a compact self-attention Python class

The Rust equivalents are `SelfAttentionV1` and `SelfAttentionV2`. `V1` stores raw tensors, making the math visible. `V2` uses `candle_nn::Linear`, making the library idiom visible. Watch out: `Linear.weight()` is exposed in `[out, in]` layout.

#### 3.5 Hiding future words with causal attention

Causal masking prevents a decoder from seeing future tokens during next-token training. This is essential for GPT-style generation but not the default for encoder classifiers.

##### 3.5.1 Applying a causal attention mask

[`get_mask(...)`][sym-get-mask] creates the mask and [`masked_fill(...)`][sym-masked-fill] applies it. Filling illegal positions with `f32::NEG_INFINITY` before softmax makes their probability effectively zero.

##### 3.5.2 Masking additional attention weights with dropout

Dropout regularizes attention weights during training. This is one reason `ModuleT` exists: train/eval behavior changes the forward pass. Repo caveat: [`CausalAttention`][sym-causalattention] is still pedagogical and hardcodes dropout in training mode; [`MultiHeadAttention`][sym-multiheadattention] exposes the real `forward_t(xs, train)` path, so call `forward_t(xs, false)` for inference if dropout is nonzero.

##### 3.5.3 Implementing a compact causal attention class

[`CausalAttention`][sym-causalattention] packages batched Q/K/V projection, mask broadcasting, softmax, dropout, and context-vector calculation into a reusable component.

#### 3.6 Extending single-head attention to multi-head attention

Multi-head attention lets the model learn multiple relevance spaces in parallel.

##### 3.6.1 Stacking multiple single-head attention layers

[`MultiHeadAttentionWrapper`][sym-multiheadattention-wrapper] is the pedagogical version: run several [`CausalAttention`][sym-causalattention] heads and concatenate outputs. It is not shape-equivalent to the efficient implementation: each wrapped head returns `d_out`, so concatenation yields `num_heads * d_out`.

##### 3.6.2 Implementing multi-head attention with weight splits

[`MultiHeadAttention`][sym-multiheadattention] is the efficient version. It projects once, reshapes into `(batch, heads, tokens, head_dim)`, computes attention for all heads, then merges back to the configured `d_out` and applies `out_proj`.

### Rust/`candle` translation

Important patterns:

- `matmul(...)` for scores and weighted sums.
- `transpose(...)` and `D::Minus1` / `D::Minus2` for axis-aware tensor math.
- `reshape(...)` plus `contiguous()` for head splitting/merging.
- `broadcast_left(...)` for mask expansion.
- `ModuleT` when dropout or train/eval behavior matters.
- [`SelfAttentionV1`][sym-selfattentionv1] and [`SelfAttentionV2`][sym-selfattentionv2] use unbatched 2D inputs, while [`CausalAttention`][sym-causalattention] and [`MultiHeadAttention`][sym-multiheadattention] switch to batched 3D inputs. If masking or matmul shapes fail, check tensor rank before checking the math.

### Why this chapter matters

Chapter 4 turns individual parts into a model. It teaches how embeddings, normalization, attention, feed-forward layers, residuals, and output heads compose into a transformer.

For the target architecture, this chapter is less about deploying [`GPTModel`][sym-gptmodel] directly and more about reading transformer model code with confidence. DeBERTa and BGE-style embedders are not this exact decoder, but they use the same family of components.

### Book anchors and learning notes

#### 4.1 Coding an LLM architecture

[`DummyGPTModel`][sym-dummygptmodel] lays out the skeleton: token embeddings, position embeddings, dropout, transformer blocks, final norm, output head. It is a wiring diagram in Rust.

#### 4.2 Normalizing activations with layer normalization

[`LayerNorm`][sym-layernorm] normalizes across the feature dimension for each token, then applies learned scale and shift. The implementation uses `mean_keepdim(D::Minus1)`, `var_keepdim(D::Minus1)`, and broadcast operations.

#### 4.3 Implementing a feed forward network with GELU activations

[`FeedForward`][sym-feedforward] widens from `emb_dim` to `4 * emb_dim`, applies [`GELU`][sym-gelu], then projects back to `emb_dim`. Attention mixes information across positions; feed-forward layers transform each position’s representation. The $4\times$ widening is a big reason the MLP contributes so many parameters, which [`X1`][x-ch04-1] in [`src/exercises/ch04.rs`][ch04-exercises] makes concrete.

#### 4.4 Adding shortcut connections

Residuals let layers learn corrections rather than full replacements. [`ExampleDeepNeuralNetwork`][sym-exampledeepnn] and [`EG05`][eg-ch04-05] make gradient-flow intuition visible.

#### 4.5 Connecting attention and linear layers in a transformer block

[`TransformerBlock`][sym-transformerblock] combines pre-norm, [`MultiHeadAttention`][sym-multiheadattention], dropout, residual, another pre-norm, [`FeedForward`][sym-feedforward], dropout, and residual. This block grammar transfers broadly across transformer families.

#### 4.6 Coding the GPT model

[`GPTModel`][sym-gptmodel] composes token embeddings, position embeddings, dropout, [`SequentialTransformers`][sym-sequentialtransformers], final norm, and an output head. [`Config`][sym-config] is the typed hyperparameter contract. This Chapter 4 implementation keeps `tok_emb` and `out_head` separate; [`EG07`][eg-ch04-07] computes a tied-weight parameter count as a comparison, not because the model already ties those weights.

#### 4.7 Generating text

`generate_text_simple(...)` performs greedy autoregressive decoding: crop to context window, forward pass, select last-position logits, softmax, argmax, append token, repeat. [`EG08`][eg-ch04-08] demonstrates the loop with a randomly initialized model, so the resulting text is expected to be nonsense until later training or pretrained-weight loading. This is excellent for learning GPT inference, but the active NL-query architecture is retrieval-first rather than generation-first.

### Rust/`candle` translation

Chapter 4 is a strong Rust design lesson:

- [`Config`][sym-config] replaces loose Python dict hyperparameters.
- [`FFLayer`][sym-fflayer] and [`SequentialTransformers`][sym-sequentialtransformers] preserve concrete type information.
- [`GPT`][sym-gpt-trait] is a minimal compatibility trait for GPT-like model variants; in Chapter 4 it only exposes `context_size()` and mainly smooths later extensions.
- `ModuleT` keeps train/eval behavior explicit. In this repo, convenience `forward()` methods on [`TransformerBlock`][sym-transformerblock] and [`GPTModel`][sym-gptmodel] call `forward_t(..., true)`, so use `forward_t(..., false)` for inference-time behavior.
- `VarBuilder::pp(...)` namespaces parameters to match model structure.

### Why this chapter matters

Chapter 5 turns a runnable model into a trainable and evaluable workflow. It teaches loss calculation, evaluation loops, training loops, sampling policies, checkpointing, and pretrained-weight loading.

For the target architecture, the generation-specific sections are mostly future-facing. But the model-engineering habits transfer directly: compute the right loss, validate on held-out data, save/load weights, and treat model artifacts as runtime contracts.

### Book anchors and learning notes

#### 5.1 Evaluating generative text models

Evaluation has qualitative and quantitative sides. Generated samples show behavior; loss shows probability assigned to correct next tokens.

##### 5.1.1 Using GPT to generate text

Generation is repeated next-token prediction: tokenize input, run the model, select next token, append, repeat. Anchors include [`text_to_token_ids(...)`][sym-text-to-token-ids], [`token_ids_to_text(...)`][sym-token-ids-to-text], and [`generate_text_simple(...)`][sym-generate-text-simple].

##### 5.1.2 Calculating the text generation loss

[`calc_loss_batch(...)`][sym-calc-loss-batch] flattens logits from `[batch, seq, vocab]` to `[batch * seq, vocab]` and targets from `[batch, seq]` to `[batch * seq]`, then applies cross-entropy. This Rust port also supports an optional `ignore_index`, filtering ignored targets before calling `candle_nn::loss::cross_entropy(...)`; later Chapter 7 collators rely on that extension. Perplexity is $e^L$.

##### 5.1.3 Calculating the training and validation set losses

`calc_loss_loader(...)` averages losses over batches. Training loss alone can flatter memorization; validation loss is the healthier signal.

#### 5.2 Training an LLM

[`train_model_simple(...)`][sym-train-model-simple] loops over batches, computes loss, performs optimizer steps, periodically evaluates, counts tokens seen, and prints sample generations. The chapter’s lesson is not just “train,” but “observe training from multiple angles.”

#### 5.3 Decoding strategies to control randomness

Model logits are not the same as a decoding policy. The same model can behave differently depending on how logits are converted into tokens.

##### 5.3.1 Temperature scaling

Temperature rescales logits before softmax: lower values sharpen, higher values flatten. It changes sampling behavior, not model knowledge.

##### 5.3.2 Top-k sampling

Top-k keeps only the highest-scoring candidates before sampling. The repo implements helper behavior in [`TopK`][sym-topk] in [`src/candle_addons.rs`][src-candle-addons], a repo-local trait implemented for `Tensor`, not a built-in `candle` abstraction. Its 2D helper includes a pragmatic CPU sorting fallback because of intermittent CUDA sorting trouble.

##### 5.3.3 Modifying the text generation function

[`generate(...)`][sym-generate] supports context cropping, optional temperature, optional top-k, a limited batch-level EOS break condition, and batched behavior. In the current implementation, generation stops early only when all batch items emit EOS on the same step, and that EOS token is not appended; it is not per-sequence early stopping. This is a more realistic inference API than the Chapter 4 greedy loop, but still a teaching implementation.

#### 5.4 Loading and saving model weights in PyTorch

The repo’s Rust equivalent is `VarMap::save(...)` and `VarMap::load(...)`. Treat `VarMap` as the named parameter store/checkpoint boundary. These calls restore model parameters only; they do not restore optimizer state, so “continuing training” resumes weights but not AdamW momentum/history.

#### 5.5 Loading pretrained weights from OpenAI

[`load_weights_into_gpt(...)`][sym-load-weights-into-gpt] maps external GPT-2 weights into the local [`GPTModel`][sym-gptmodel]: names must match, shapes must match, transposes may be needed, and QKV layouts must be handled carefully. Pretrained loading is tensor bookkeeping, not magic.

### Rust/`candle` translation

Important patterns:

- `ModuleT` for training-aware forward passes.
- `AdamW` and `Optimizer` for updates.
- `VarMap` for save/load boundaries.
- `hf_hub` and `safetensors` for pretrained artifacts.
- `Option` for decoding knobs such as temperature, top-k, EOS, and ignore-index behavior.

The backbone differs, but the workflow is similar: labeled data, tokenization, model initialization, classification head, supervised loss, validation, checkpointing, and inference.

### Book anchors and learning notes

#### 6.1 Different categories of fine-tuning

Fine-tuning is a spectrum: head-only, top-layers, or full-model. The repo’s main path trains the final transformer block, final norm, and output head. [`X2`][x-ch06-2] in [`src/exercises/ch06.rs`][ch06-exercises] explores full-model fine-tuning.

#### 6.2 Preparing the dataset

The repo downloads, balances, and splits SMS spam data. The transfer is pipeline discipline: inspect labels, handle imbalance intentionally, and keep train/validation/test boundaries clean.

#### 6.3 Creating data loaders

[`SpamDatasetBuilder`][sym-spamdatasetbuilder], [`SpamDataset`][sym-spamdataset], [`SpamDatasetIter`][sym-spamdatasetiter], and [`SpamDataLoader`][sym-spamdataloader] translate dataframe rows into padded token tensors plus labels. This is the closest repo analogue to building a labeled query-routing dataset.

#### 6.4 Initializing a model with pretrained weights

[`download_and_load_gpt2(...)`][sym-download-and-load-gpt2] starts from pretrained GPT-2 weights. The target router similarly starts from a pretrained DeBERTa sequence-classification model rather than training language understanding from scratch.

#### 6.5 Adding a classification head

`modify_out_head_for_classification(...)` replaces a vocabulary-sized language-model head with a class-sized output head. Spam uses two classes. The target router uses three: `graph_search`, `vector_search`, `graph_rag`.

#### 6.6 Calculating the classification loss and accuracy

The repo computes class logits, cross-entropy loss, argmax predictions, and accuracy. GPT classification uses a selected token position. In this repo, the default is the final sequence slot after padding, not necessarily the last semantic word; that works as a GPT-specific teaching choice because the hidden state can attend to the prefix. DeBERTa classification uses an encoder sequence-classification head or pooled representation, so do not copy the last-token trick blindly.

Shape-wise, the Chapter 6 classifier path is: inputs `[batch, seq]`, model outputs `[batch, seq, classes]`, select one sequence index to get `[batch, classes]`, then compare with labels `[batch]`. For a production three-class query router, extend the repo’s minimal loss/accuracy loop with macro F1, per-class precision/recall, confusion matrices, calibrated confidence scores, and explicit abstain/fallback threshold evaluation. No target-router accuracy should be inferred from the Chapter 6 spam task.

#### 6.7 Fine-tuning the model on supervised data

`train_classifier_simple(...)` trains and evaluates the classifier. [`EG13`][eg-ch06-13] shows selective parameter training by filtering `VarMap` variable names such as `final_norm`, `out_head`, and `trf.11`.

#### 6.8 Using the LLM as a spam classifier

[`classify_review(...)`][sym-classify-review] is the narrow inference API: raw text in, tokenization/padding/model forward/class label out. For routing, online inference must use the same truncation and padding policy as training; the current helper is educational and does not fully model a production router’s input-contract checks. The target router needs the same narrowness, plus softmax probabilities and low-confidence fallback policy.

For the current NL-query architecture, this is not the active request path: Phase 11 is retrieval-only. But the chapter is highly relevant for future local generation, answer shaping, structured output, or planner-style model components.

### Book anchors and learning notes

#### 7.1 Introduction to instruction fine-tuning

Instruction fine-tuning is still supervised next-token learning. The main change is the training text format: instruction + optional input + response. Do not treat it as a new magic objective.

#### 7.2 Preparing a dataset for supervised instruction fine-tuning

[`InstructionResponseExample`][sym-instructionresponseexample] stores instruction, optional input, output, and later optional model response. The core book path is Alpaca-format instruction tuning; repo exercise extensions explore prompt-style swaps, response-only masking, Alpaca-scale data, and LoRA experiments. [`PromptFormatter`][sym-promptformatter], [`AlpacaPromptFormatter`][sym-alpacaformatter], and [`Phi3PromptFormatter`][sym-phi3formatter] demonstrate formatting policy separated from data storage, though the Phi-3 path is an exercise-oriented extension and not a clean drop-in equivalent in the current code.

For NL-query work, the dataset is your interface specification. If you want JSON, SQL, a plan object, or a clarification request, that output shape must appear consistently in training examples.

#### 7.3 Organizing data into training batches

[`InstructionDataCollator`][sym-instructiondatacollator] pads variable-length examples and builds shifted targets. Concretely, it shifts each sequence left, appends one terminal `pad_token_id`, and marks only the extra batch-padding positions with `DEFAULT_IGNORE_INDEX`; the core collator masks padding, not instruction tokens. Response-only masking appears in exercises, where the collator additionally masks the instruction/input prefix so loss focuses on response tokens.

#### 7.4 Creating data loaders for an instruction dataset

[`InstructionDatasetIter`][sym-instructiondatasetiter], [`InstructionDataBatcher`][sym-instructiondatabatcher], and [`InstructionDataLoader`][sym-instructiondataloader] make variable-length instruction examples trainable. The repo implements a custom batcher because default batching assumptions do not fit ragged sequences before collation.

#### 7.5 Loading a pretrained LLM

Chapter 7 reuses [`download_and_load_gpt2`][sym-download-and-load-gpt2] from Chapter 6. This reinforces the pattern: pretrained model first, task/interface adaptation second.

#### 7.6 Fine-tuning the LLM on instruction data

[`EG10`][eg-ch07-10] loads GPT-2, gets loaders from [`EG07`][eg-ch07-07], creates `AdamW`, calls [`train_model_simple`][sym-train-model-simple], and saves `ift.checkpoint.safetensors`. The training loop is reused from Chapter 5; Chapter 7 mainly changes the data contract.

#### 7.7 Extracting and saving responses

`generate_test_set_responses(...)` runs the fine-tuned model on held-out examples and saves `model_response` values to JSON. Durable artifacts make repeatable evaluation possible.

#### 7.8 Evaluating the fine-tuned LLM

The repo uses loss, manual output inspection, and a local Ollama-based judge through `query_model(...)` and `generate_model_scores(...)`. For structured NL-query work, add deterministic validators such as parse success, schema validity, execution success, and result equivalence.

#### 7.9 Conclusions

The model architecture barely changes. The training objective barely changes. The dataset and prompt contract change a lot. That is the hidden lesson of instruction tuning: output behavior is shaped by interface examples.

##### 7.9.1 What’s next?

Next steps depend on the product problem: stronger supervised examples, response-only masking, LoRA for efficient adaptation, or DPO for preference alignment.

##### 7.9.2 Staying up to date in a fast-moving field

Anchor to principles: dataset contract, masking, saved artifacts, validators, and controlled evaluation age better than model-name trends.

##### 7.9.3 Final words

Pretraining gives the model broad language ability. Instruction tuning gives it a job description.

### Why it matters

LoRA teaches adaptation without updating every original model weight. Full fine-tuning says “move everything.” LoRA says “freeze the base model and learn a small low-rank correction.” That matters for future local adaptation, especially when model size or hardware budget makes full fine-tuning unattractive.

For the active NL-query architecture, this is not a current request-path requirement. But it is a valuable rehearsal for adapting local classifiers or future local generators in Rust.

### Book anchors and learning notes

#### E.1 Introduction to LoRA

LoRA learns a low-rank correction to a weight matrix. Instead of learning a full $\Delta W$, it learns two smaller matrices whose product forms the update:

$$
W' = W + \alpha BA
$$

[`LoRALayer`][sym-loralayer] stores `A`, `B`, and `alpha`. In this port, `A` has shape `[rank, in_dim]` and `B` has shape `[out_dim, rank]` to match `candle` linear weights, and the LoRA path contributes `alpha * x A^T B^T`, an effective `alpha * BA` update. Unlike the common $\alpha / r$ formulation, this implementation applies `alpha` directly. `B` starts at zero, so a freshly wrapped layer initially behaves like the base layer.

#### E.2 Preparing the dataset

Appendix E reuses the Chapter 6 spam-classification data path. That keeps the spotlight on adaptation strategy rather than introducing a new task. Most of Appendix E’s new work is adapter wrapping and trainable-parameter selection; dataset prep, GPT loading, and classifier training are mostly Chapter 6 machinery reused for a LoRA-specific experiment.

#### E.3 Initializing the model

The repo loads pretrained GPT-2, replaces the output head for classification, and then builds LoRA-aware wrappers. This port also wraps the replaced classification `out_head` with LoRA, so the adapterized surface is broader than the “attention-only” LoRA pattern some readers may expect. The most important `candle` reality: `replace_linear_with_lora(...)` is documented as a no-op. There is no generic reflective module-tree replacement here.

Instead, the repo implements explicit wrappers:

- [`LinearWithLoRA`][sym-linearwithlora]
- [`MultiHeadAttentionWithLoRA`][sym-mha-with-lora]
- [`FeedForwardWithLoRA`][sym-feedforward-with-lora]
- [`TransformerBlockWithLoRA`][sym-transformerblock-with-lora]
- [`SequentialTransformersWithLoRA`][sym-sequentialtransformers-with-lora]
- [`GPTModelWithLoRA`][sym-gptmodelwithlora]

#### E.4 Parameter-efficient fine-tuning with LoRA

The training loop is mostly ordinary classifier fine-tuning. The difference is optimizer parameter selection: base weights are effectively frozen by optimizer scope, because training collects only LoRA `A` and `B` variables from `VarMap` and passes those to `AdamW`. Parameter-efficient fine-tuning is the same forward/training spine with a smaller trainable set, though the examples save and reload the full `VarMap` rather than adapter-only checkpoints.

### Why it matters

Supervised instruction tuning teaches a model to imitate a target response. DPO teaches a model to prefer one response over another. That distinction matters when correctness is not the only product concern and style, helpfulness, refusal behavior, or explanation quality need alignment.

The current NL-query architecture does not need DPO for its retrieval-first baseline. It becomes relevant if a later phase adds local answer generation or user-facing response composition.

### Core ideas

A DPO example contains a prompt, a chosen response, and a rejected response. The trainable policy model and frozen reference model both score the chosen and rejected completions. The loss rewards the policy for preferring the chosen response more strongly than the reference does.

Compactly:

$$
\mathcal{L}_{DPO}
=
-\log \sigma\left(
\beta\left[
(\log \pi_\theta(y^+|x) - \log \pi_\theta(y^-|x))
-
(\log \pi_{ref}(y^+|x) - \log \pi_{ref}(y^-|x))
\right]
\right)
$$

Plain English: make the policy prefer the better answer over the worse answer relative to a frozen baseline.

Repo-specific note: `compute_logprobs(...)` scores a completion by the average log-probability over masked completion tokens after the one-token label shift. That makes this implementation a length-normalized variant of the textbook DPO objective rather than a raw summed sequence log-probability.

- [`PreferenceExample`][sym-preferenceexample]

- [`PreferenceDataLoader`][sym-preferencedataloader]

The collator is the heart of the learning path. It builds separate chosen/rejected tensors, pads/truncates them with a shared batch-wide length policy, and masks prompt/padding tokens so the preference loss focuses on completion regions. Its default length cap is `allowed_max_length: Some(1024)`.

In [`07.23`][eg-ch07-23], the policy and reference are loaded from the same instruction-finetuned checkpoint into separate `VarMap`s; only policy variables are passed to `AdamW`, which keeps the reference frozen.

- Attention and transformer component literacy from Chapters 3–4.
- Model artifact loading and parameter management from Chapters 5–6.
- Classification adaptation workflow from Chapter 6.
- Dataset/iterator/collator/loader patterns from Chapters 2, 6, 7, and DPO.

### What transfers as mental model only

- GPT-specific causal masking.
- Autoregressive generation loops.
- Temperature and top-k sampling.
- Instruction tuning for a future answer generator.
- DPO for preference alignment.

### What is mostly future work

- Local answer generation on top of retrieved context.
- Parameter-efficient adaptation with LoRA.
- Preference optimization with DPO.
- Combining LoRA and DPO.
- LLM-judge evaluation beyond deterministic validators.

### Decoder repo vs encoder target architecture

This repo is decoder-heavy because the book teaches GPT from scratch. The target architecture is encoder-heavy because its active ML components classify and embed. That mismatch is not a reason to ignore the repo. It is a reason to transfer carefully:

- Transfer tensors, tokenization, shapes, checkpoints, attention, and classifier adaptation.
- Do not over-transfer GPT causal generation into a retrieval-first system.

### `candle` lessons that matter in production Rust

- Explicit parameter names matter.
- `ModuleT` train/eval behavior matters.
- Device placement is a design choice.
- Collators are architecture-critical.
- Library gaps are normal; small local utilities such as [`TopK`][sym-topk] and [`SequentialT`][sym-sequentialt] are part of real adaptation work.
- Manual wrapper types can be verbose, but they reveal model structure clearly.
